INTERPRETATION

- The draft model nearly doubled the aggregate median decode rate at all three
  input lengths, with gains from +90.3% to +105.8%.
- The 45.39% token acceptance rate produced a mean accepted sequence of 3.27
  tokens per target-model verification step.
- Structured and predictable tasks benefited most. Copying, code completion,
  field extraction, and the arithmetic prompt gained approximately 90% to 149%.
- Benefits were not determined solely by the assigned cohort. Field extraction
  and arithmetic performed very well, while formal rewriting and creative
  writing slowed down.
- The creative prompt lost 4% to 21% decode performance. Its draft proposals
  did not offset the cost of running the additional 0.6B model.
- Draft-model TTFT was higher than control by approximately 47 ms at 256 tokens,
  83 ms at 2,048 tokens, and 135 ms at 8,000 tokens.
- Draft-model speculation therefore trades higher first-token latency for
  faster generation. It is most useful when responses are long enough for
  decode savings to recover the additional TTFT.
- Peak observed VRAM was 41,481 MiB. The draft weights loaded without OOM on
  the 48 GB L40S at the 0.90 memory-utilization setting.

VALIDITY CRITERIA

- The server launch log confirms Qwen3-0.6B with five speculative tokens.
- All three result files record label=draft, the correct draft model, and depth 5.
- Each run completed 27 requests with no reported benchmark error.
- Every request used an exact input length of 256, 2,048, or 8,000 tokens.
- Control and draft runs used the same prompts, output limit, temperature,
  concurrency, and prefix-caching policy.
- The first request after server startup showed a warm-up TTFT outlier. Three
  runs and cohort medians prevent it from determining the reported result.

LIMITATIONS

- This is a sequential latency benchmark, not a concurrent throughput test.
- The overall median gives every prompt equal weight and may not match a real
  production workload mix.
- Some completions ended before 256 output tokens because 256 was a maximum,
  not a forced output length.
- Very short completions make decode-rate measurements more sensitive to
  timing overhead.
- Speculative acceptance metrics are aggregated across each complete run and
  are not available per prompt or cohort.
- Draft-model speculation forced vLLM's V1 runner and disabled async scheduling,
  so the result includes those implementation differences from control.
