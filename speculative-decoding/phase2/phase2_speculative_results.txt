PHASE 2 — QWEN3-32B AWQ SPECULATIVE DECODING COMPARISON

Status: Complete
Run date (UTC): 2026-07-19

OBJECTIVE

Compare a normal vLLM control against n-gram prompt lookup at depths 2, 4, and
8, plus Qwen3-0.6B draft-model speculation with five proposed tokens.

CONFIGURATION

Target model: Qwen3-32B AWQ (INT4)
Draft model: Qwen3-0.6B (FP16)
Serving engine: vLLM 0.25.1
GPU: 1x NVIDIA L40S, 46,068 MiB
Tensor parallel size: 1
Maximum model length: 16,384 tokens
GPU memory utilization setting: 0.90
Input lengths: 256, 2,048, and 8,000 tokens
Maximum output length: 256 tokens
Prompts per input length: 9
Prompt cohorts: high reuse, medium reuse, and low reuse (3 prompts each)
Runs per mode: 3
Requests per mode: 81
Total measured requests: 405
Concurrency: 1 (sequential)
Temperature: 0.0
Thinking mode: disabled
Prefix caching: disabled

MODES

control: No speculative decoding
ngram-2: Prompt lookup with 2 speculative tokens
ngram-4: Prompt lookup with 4 speculative tokens
ngram-8: Prompt lookup with 8 speculative tokens
draft: Qwen3-0.6B proposing 5 tokens for Qwen3-32B verification

KEY FINDINGS

1. There is no universally best speculative method.

   Ngram-8 dominated the high-reuse cohort, while the draft model had the best
   aggregate median and performed well on extraction and arithmetic. Control
   remained best for the medium-reuse cohort at 8,000 tokens.

2. N-gram performance is highly dependent on reusable token sequences.

   High-reuse speedups increased with n-gram depth and reached +287.8%.
   Medium-reuse prompts slowed by approximately 6% to 8% at every depth.

3. Lower acceptance rate does not necessarily mean lower throughput.

   Ngram-8 had the lowest token acceptance rate at 23.47%, yet it delivered the
   fastest n-gram decode performance because successful proposals accepted
   longer sequences and high-reuse prompts benefited from deeper lookahead.

4. The draft model produced broader but still uneven gains.

   Its aggregate median decode speed improved by 90% to 106%. Copying, code,
   extraction, and arithmetic gained strongly, while creative writing and
   formal rewriting slowed down.

5. Speculation affects TTFT differently by method.

   N-gram TTFT was about 16 ms above control at 256 tokens and nearly unchanged
   at longer contexts. Draft-model TTFT was approximately 47 to 135 ms higher
   because the additional model and V1 speculative path add overhead.

6. Workload routing is more promising than enabling one method globally.

   Based on this small prompt set, ngram-8 is a strong choice for copying and
   repeated/templated output. Draft speculation is attractive for extraction,
   arithmetic, and longer structured responses. Control may be preferable for
   creative or rewrite workloads with poor speculative efficiency.

OPERATIONAL TRADE-OFFS

N-gram:
- Requires no additional model weights or training.
- Adds little TTFT at medium and long input lengths.
- Can substantially slow workloads without useful token reuse.
- Deeper lookup increases proposal work and lowers token acceptance rate.

Draft model:
- Requires loading and executing an additional 0.6B model.
- Used approximately 41,481 MiB peak VRAM and fit on the L40S without OOM.
- Forced vLLM's V1 runner and disabled async scheduling in vLLM 0.25.1.
- Added more TTFT than n-gram speculation.
- Delivered the highest aggregate decode medians in this benchmark.

VALIDITY AND PROVENANCE

- Every mode used three complete benchmark runs.
- Each mode completed 81 requests: 9 prompts at 3 input lengths across 3 runs.
- All 405 measured requests completed without a benchmark error.
- Every request used an exact input length of 256, 2,048, or 8,000 tokens.
- Prefix caching was disabled to prevent cross-request prefill reuse.
- Temperature, output limit, prompts, context lengths, and concurrency were
  held constant.
- Server launch records were checked against result labels and configuration.
- Three initial ngram-4 files were mislabeled ngram-2 during collection. Their
  server record proved depth 4 was active, and both filenames and internal
  labels were corrected before analysis.
- Stale embedded depth metadata in the later ngram-2 and ngram-8 JSON files was
  corrected to match their verified server launch configurations.
- Draft filenames and embedded configuration matched the draft server record
  without correction.

LIMITATIONS

- This is a sequential latency benchmark, not a load test. It does not measure
  aggregate throughput, queueing, or behavior with concurrent users.
- The nine prompts are a small synthetic workload and do not establish a
  universal production routing policy.
- Cohort labels describe expected reuse, but individual prompts within a
  cohort behaved differently.
- Some responses stopped before 256 output tokens because 256 was a maximum,
  not a forced output length.
- Very short completions make decode-rate measurements sensitive to timing
  overhead.
- Acceptance counters are aggregated per run, not per prompt or cohort.
- The first request after each speculative server launch included warm-up
  overhead. Three runs and cohort medians reduce its influence.
- End-to-end user latency depends on both TTFT and output length; decode tok/s
  alone does not determine which mode is best for short responses.
